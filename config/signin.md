# 登录保持优化记录

## 背景

最初的登录恢复主要依赖 Electron/Chromium 自己保存的 session cookie。实测发现，关闭程序重新打开后有时可以恢复，但过几个小时后仍可能提示 `not logged in or login timeout`。

原因是网络学堂的登录态可能包含服务端会话时效；浏览器登录产生的 session cookie 也不一定能长期由 Chromium 自动恢复。只保存 cookie 不能保证服务端会话永不过期。

## 已做优化

1. 登录成功后，将 API session 中的安全 cookie 用 Windows `safeStorage` 加密保存到 `session.enc`。
2. 启动时先恢复 `session.enc`，再调用真实接口验证是否仍然有效。
3. 如果用户使用账号密码登录并勾选了“记住密码”，启动时会把加密保存的账号密码放入内存，后续接口遇到登录过期可自动重登一次。
4. 程序运行期间增加低频保活，每 10 分钟轻量探测一次 session；如果失效且存在已保存账号密码，会自动重登。
5. 退出登录时会同时清理保存的 cookie 和账号密码，避免旧凭据继续生效。

## 当前边界

浏览器登录依赖清华统一身份认证或 WebVPN 的网页流程。如果用户只使用浏览器登录、没有保存账号密码，服务端会话过期后客户端无法静默完成二级验证，只能提示用户重新登录。

因此最稳定的“长时间免登录”方式是：在可用环境下使用账号密码登录，并勾选“记住密码”。浏览器登录仍保留为推荐入口，适合需要 WebVPN、二级验证或校园网认证的场景。
