# 作业提交修复记录

## 背景

作业详情页可以选择附件，但点击“确认提交”后一直失败。最开始服务端提示“请上传附件”，后续本地校验提示“附件为空，请重新选择文件”。

## 主要原因

提交附件这条链路有两类风险：

1. 渲染进程在 Electron sandbox 下不能可靠直接读取本地文件，只能把路径交给主进程。
2. 用户选择的文件可能来自云盘占位、临时位置或被移动，导致提交时主进程读到的是空文件或不可用文件。
3. 手写 multipart 必须尽量贴近网络学堂原始表单字段，否则服务端可能认为 `fileupload` 字段为空。

## 修复思路

1. 选中文件后，主进程立即读取并复制到本地临时提交目录。
2. 复制时校验文件是否存在、是否为文件、内容是否非空。
3. 前端展示的是已校验副本的文件名和大小，避免“看起来选了，提交时才失败”。
4. 提交时使用临时副本路径，不再依赖原始文件位置。
5. multipart 字段顺序对齐 `thu-learn-lib`：`xszyid`、`zynr`、`fileupload`、`isDeleted`。
6. 提交请求使用 Node HTTPS，并显式设置 `Content-Length`。

## 当前结果

附件在选择阶段就会被校验和固化。只要界面显示已选择且带有正常大小，提交阶段应能读取到真实文件内容。

如果仍然失败，下一步优先检查服务端是否要求先进入提交页面刷新 CSRF 或是否对某些作业类型有额外必填字段。

## 后续补充

已提交但未截止的作业详情页现在会显示“修改提交”区域，允许继续更新文本或重新选择附件。我的提交内容会按 HTML 消毒后展示，避免把网络学堂富文本编辑器的隐藏标签、注释或空 `span` 当成代码显示。
